> ほほ笑みラボの生徒さん向け多人数ゲームを、**1つのアプリで複数選べる**形に統合する。
> 既存の7並べ（Socket.io・LAN型）を土台に、神経衰弱を同じ基盤へ載せる。
> **統合の主目的**：1つのアプリで「7並べ」「神経衰弱」を選んで遊べるようにする。

## 1. 統合の基本方針

### 1.1 「共有プラットフォーム＋ゲームモジュール」方式
共通部分を土台（プラットフォーム）として切り出し、各ゲームを差し込み式のモジュールにする。

```
┌─────────────────────────────────────────────┐
│  共有プラットフォーム                         │
│  ・トップ画面 / ゲーム選択                    │
│  ・部屋作成・合言葉入室・QRコード・席割り      │
│  ・Socket.io 通信基盤（LAN・ホストPC型）      │
│  ・音声・効果音の再生基盤                     │
│  ・シニア向けUI部品（大きい文字・高コントラスト）│
│  ・CPU基盤（弱・中・強の枠組み）              │
│  ・結果／ランキング画面の体裁                 │
├─────────────────────────────────────────────┤
│  ゲームモジュール（差し込み式）               │
│  ┌─────────────┐   ┌─────────────┐          │
│  │ 7並べモジュール│   │神経衰弱モジュール│          │
│  │（既存・流用） │   │（新規・移植） │          │
│  └─────────────┘   └─────────────┘          │
└─────────────────────────────────────────────┘
```

### 1.2 ゲームモジュールの考え方
各ゲームは「土台に対して以下を宣言・提供する」モジュールとして実装する。
- ゲーム固有のルールロジック（純粋TS）
- ゲーム固有のUI（盤面・手札・操作）
- ゲーム固有の通信イベント
- **プレイヤールール**（最小／最大人数、CPU可否、CPU補完の要否）
- ゲーム固有のCPU思考（土台の弱・中・強の枠に沿って実装）

土台側は「どのゲームか」を意識せず、ロビー・部屋・通信・音声・席割りを共通提供する。これにより、**7並べのロジックには手を入れずに**、神経衰弱を追加できる。

## 2. 通信基盤（Socket.io・LAN完結を維持）

### 2.1 方針
- **教室内LAN完結・インターネット不要**を維持（Supabaseは使わない）。
- ホストPCのNodeサーバー上で Next.js ＋ Socket.io を同居（既存7並べと同じ）。
- 各端末はQRコード／URL＋4桁合言葉で入室。

### 2.2 神経衰弱の「カンニング防止」をLAN型で実現する
神経衰弱は伏せ札の中身を覗けるとゲームが成立しない。元企画書はこれをSupabase（サーバ側のみが中身を保持）で実現する想定だったが、**Socket.io型ではホストPCのサーバーがその役割を担う**。

- **カードの表（中身）はホストPCサーバーだけが保持**する。
- 各プレイヤーの端末には、伏せ札の「位置」と「伏せ状態」だけを配り、**中身は送らない**（ブラウザを覗いてもカンニングできない）。
- めくり操作は「この位置をめくりたい」という要求をサーバーに送り、サーバーが中身を判定して結果を返す。
- 「覗き見」効果も同様に、**発動者の端末にだけ**サーバーが中身を一定時間返す（他端末には送らない）。

> これにより、クラウドに出さずにLAN内で神経衰弱の本質的要求（中身の秘匿）を満たせる。

---

## 3. ゲーム選択フロー

**一つの部屋＝一つのゲーム**（シンプル方針）。

1. ホストがトップ画面で「部屋を作る」
2. **ホストが遊ぶゲームを選択**（7並べ / 神経衰弱）
3. 選んだゲームの設定を行う（後述§4の各ゲーム設定）
4. 合言葉とQRコードを表示
5. 参加者が入室 → 席／プレイヤー登録
6. ホストが「CPUで埋める」（人数不足時）→「対局開始」
7. 選択したゲームのモジュールが起動
8. 終了 → 結果画面 →「もう一回（同じゲーム）」or「部屋を解散」

- 部屋の途中でゲームを切り替えることはしない（別ゲームをやりたければ部屋を作り直す）。

## 4. プレイヤー／CPUルール（ゲームごとに宣言）

| 項目 | 7並べ | 神経衰弱 |
|------|-------|----------|
| 人数 | 4人固定 | 2〜4人（可変） |
| CPU | あり（4人になるよう補完） | **あり**（人数が足りないとき補完） |
| CPU強さ | 弱・中・強 | 弱・中・強（**記憶保持率ベース**／後述） |

- 土台の「CPU基盤（弱・中・強の枠組み）」を両ゲームで共有する。
- ただしCPUの**思考の中身はゲームごとに別実装**（7並べ＝出せる札の戦略、神経衰弱＝記憶と選択）。

## 5. 神経衰弱モジュール（Socket.io移植版）

> 元企画書（`REQUIREMENTS_concentration.md`）のゲーム設計をそのまま踏襲し、
> **技術基盤のみ Supabase → Socket.io に置き換え**、さらに**CPUを追加**する。

### 5.1 ゲーム概要
- 場に伏せて並べたカードから、順番に2枚めくる。
- **同じ絵柄のペアが揃えば**獲得し、**続けてもう一度めくれる**。
- **揃わなければ**2枚を伏せに戻し、次の人の番へ。
- 場のカードが無くなったら終了。**合計点が最も高い人が勝ち**（枚数でなく点数）。

### 5.2 企画の狙い（平等な神経衰弱）
通常の神経衰弱は記憶力が強い人が勝ち続ける。**暗記が得意でなくても勝つチャンスがある**ようにするため、以下3つを導入する。狙いは「記憶の有利を消す」ではなく「記憶が得意でも独占させない」。

### 5.3 盤面（枚数モード）
- **フル**：52枚／26ペア（じっくり長め）
- **教室モード**：ペア数を絞る（初期値 12〜15ペア程度。短時間・シニアにやさしい）
- ペア数は**設定可能な定数**（モードはプリセット）。ペア数が少ないほど運の比重が上がる＝難度調整ツマミ。

### 5.4 特殊カード（A方式・即発動／3種）
- 特殊カードは**ペアにならない単独カード**で、通常カードに混ぜて伏せておく。
- **めくった瞬間に効果が即発動**。発動後、**その手番は終了して次の人へ**（連続めくり無し＝諸刃の剣）。
- 発動したら場から除外（使い切り）。
- 枚数：全体の **1〜2割** が目安（設定可能）。

| 種類 | 効果 |
|------|------|
| **シャッフル** | 場に残る伏せカードのうち2〜3組（4〜6枚）の位置を入れ替える（部分シャッフル）。全員に「シャッフルします！」と予告し、対象を数秒見せてから伏せる演出。枚数は設定可能 |
| **入れ替え** | 発動者が好きな伏せカード2枚を選び位置を入れ替える（自分は元位置を覚えている前提＝頭脳系）。他プレイヤーには「2枚が入れ替わった」事実のみ通知 |
| **覗き見** | 発動者が好きな伏せカード1枚を選び、**自分の画面でだけ**中身を一時的に確認（数秒）。他端末には中身を送らない。1回の発動で1枚のみ（サーバー側で厳密に制限） |

### 5.5 得点（点数差ペア）
- 獲得は枚数でなく**点数**で競う。
- **ほとんどのペアは1点**。ただし**数組だけ2〜3点の高得点ペア**を混ぜる（高得点ペアは全ペアの1〜2割程度。設定可能）。
- どのペアが高得点かは伏せた状態では分からない（揃えて判明）。
- 最終的に**合計点が最大のプレイヤーが勝利**。同点時は獲得ペア数が多い方を上位とする。

### 5.6 神経衰弱のCPU（記憶保持率ベース）
**設計上の重要点**：記憶が完璧なCPUは無敵になってしまう（めくられた全カードを覚えて必ず揃える）。そこで**あえて不完全に忘れるCPU**にする。これは「平等な神経衰弱」の狙いとも合致する。

| 強さ | 記憶保持の挙動 |
|------|----------------|
| 弱 | 直近に見たごく少数の札しか覚えていない／高確率で忘れる。生徒さんが勝ちやすい |
| 中 | そこそこ覚えるが、時間が経つと忘れる。たまにミスする |
| 強 | 多くの札を覚えるが**完璧ではない**（保持率100%にはしない）。揃えられるときは積極的に揃える |

- CPUが特殊カードをめくった場合もA方式で即発動：
  - シャッフルは自動発動。
  - 入れ替え・覗き見は、CPUが妥当な対象を自動選択（例：覗き見は未確認の札を1枚選ぶ等）。
- CPUの思考時間：1〜2秒（待たせすぎない演出）。

### 5.7 画面・表示
- 場：伏せカードのグリッド表示。めくり中の2枚をハイライト。
- 各プレイヤーの**現在の合計点**と手番表示。
- 揃ったペアの**点数を一瞬表示**（1点／2点／3点が分かる）。
- 特殊カード演出：
  - シャッフル → 「シャッフルします！」予告 → 対象を数秒オープン提示 → 入れ替えて伏せる、を**全員の画面で同期**。
  - 覗き見 → 発動者だけ中身が見える。他は「覗き見発動」表示のみ。
- 結果画面（最終点数の順位）。土台の体裁に合わせる。

## 7. 技術スタック・ディレクトリ構成

### 7.1 技術スタック
- Next.js（App Router）+ TypeScript + Tailwind CSS
- Zustand（状態管理）
- Socket.io（クライアント・サーバー、LAN・ホストPC型）
- 音声：HTMLAudioElement / Web Audio API（事前録音mp3）
- カード素材：SVG自作（`public/cards/` を両ゲームで共有）

### 7.3 ゲームモジュール共通インターフェース（イメージ）
`lib/platform/gameModule.ts` で各ゲームが満たす型を定義する。
- `id` / `name`（"sevens" / "concentration"）
- `minPlayers` / `maxPlayers` / `cpuFill`（CPU補完の要否）
- `createInitialState(config)` （初期状態生成）
- `handleAction(state, action)` （行動処理）
- `getView(state, playerId)` （プレイヤーごとの可視状態。**神経衰弱はここで中身を隠す**）
- `isFinished(state)` / `getRanking(state)`

## 8. 開発ロードマップ

### フェーズ1：土台の切り出し（7並べを壊さない）

### フェーズ5（将来）
- ゲームの追加（坊主めくり等）も同じモジュール方式で。
- クラウド対戦への拡張（必要になれば）。

## 9. やらないこと（MVP・先読み禁止）

### プラットフォーム共通
- ❌ 部屋の途中でのゲーム切り替え（一部屋＝一ゲーム）
- ❌ クラウド化・インターネット越し対戦（LAN完結を維持）
- ❌ 認証・ランキング永続化・課金
- ❌ 対局記録の保存（その場限り）
- ❌ **既存7並べのルール変更**（載せ替えのみ、挙動は変えない）

### 神経衰弱
- ❌ ジョーカー／追加の特殊効果（5.4の3種のみ）
- ❌ ハンデ機能・チーム戦・ラバーバンド（将来検討）
- ❌ 効果の連鎖や複雑な相互作用（特殊カードはめくった本人にのみ即時作用）
- ❌ 設定可能と書いていない値の追加パラメータ化

## 10. 確定仕様サマリ

- **統合方式**：共有プラットフォーム＋ゲームモジュール（7並べは載せ替えのみで壊さない）
- **通信**：Socket.io・LAN完結（Supabaseは使わない）。神経衰弱の中身秘匿はホストPCサーバーで実現
- **ゲーム選択**：部屋作成時にホストが選ぶ（一部屋＝一ゲーム）
- **7並べ**：4人固定／CPU弱中強／標準52枚・ジョーカーなし（既存仕様）
- **神経衰弱**：2〜4人／CPUあり（記憶保持率ベース・あえて不完全）／特殊カードA方式3種／点数差ペア／枚数モード選択
- **共有素材**：トランプSVG・音声・シニアUI部品
